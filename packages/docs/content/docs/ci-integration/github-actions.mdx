title: GitHub Actions
description: Set up GitHub Actions webhook events for Citric.
---

Citric receives GitHub Actions telemetry via webhooks. You can set this up using a **repository webhook** or a **GitHub App** (recommended).

Both methods require subscribing to two webhook events:
- **Workflow jobs** (`workflow_job`)
- **Workflow runs** (`workflow_run`)

## Repository webhook

Create a webhook on a specific repository. You need repository owner or admin access.

1. Navigate to your repository on GitHub
2. Go to **Settings** > **Webhooks** > **Add webhook**
3. Set **Payload URL** to your collector endpoint (e.g. `https://collector.example.com/webhook/github`)
4. Set **Content type** to `application/json`
5. Enter a **Secret** — a random high-entropy string. Configure the same value in your collector config
6. Under **Which events would you like to trigger this webhook?**, select **Let me select individual events**
7. Uncheck **Pushes** (selected by default)
8. Check **Workflow jobs** and **Workflow runs**
9. Ensure **Active** is checked
10. Click **Add webhook**

You can also create webhooks via the [GitHub REST API](https://docs.github.com/en/rest/webhooks/repos#create-a-repository-webhook) or with Terraform using the [`github_repository_webhook`](https://registry.terraform.io/providers/integrations/github/latest/docs/resources/repository_webhook) resource.

## GitHub App (recommended)

GitHub Apps have higher rate limits, support multiple repositories from a single installation, and provide finer-grained permissions. If you plan to use GitHub API authentication for log fetching, a GitHub App is the recommended approach.

### Create the app

1. Go to [Settings > Developer Settings > GitHub Apps](https://github.com/settings/apps)
2. Click **New GitHub App**
3. Choose a name and enter a Homepage URL
4. Uncheck **Expire user authorization tokens**
5. Under **Webhook**, ensure **Active** is checked
6. Set the **Webhook URL** to your collector endpoint (e.g. `https://collector.example.com/webhook/github`)
7. Enter a **Webhook secret** — a random high-entropy string
8. Under **Permissions > Repository permissions**, set **Actions** to **Read-only**
9. Under **Subscribe to events**, check **Workflow job** and **Workflow run**
10. Select where the app can be installed
11. Click **Create GitHub App**
12. Note the **App ID** shown at the top of the settings page
13. At the bottom of the page, click **Generate a private key** and save the `.pem` file

### Install the app

1. Go to [Settings > Developer Settings > GitHub Apps](https://github.com/settings/apps) and click **Edit** on your app
2. In the left sidebar, click **Install App**
3. Click **Install** next to the target account
4. Select which repositories to install on
5. Click **Install**
6. Go back to the app settings > **Advanced** > **Recent deliveries**
7. Expand a delivery and confirm the payload includes `installation.id`

### Configure the collector

Add the App ID and private key path to your collector config:

```yaml
receivers:
  githubactions:
    endpoint: 0.0.0.0:3333
    path: /webhook/github
    secret: your-webhook-secret
    gh_api:
      auth:
        app_id: 1234
        private_key_path: /path/to/key.pem
```

> The receiver resolves installation IDs from each webhook payload (`installation.id`), so one receiver instance can serve multiple app installations.

## Verifying the setup

After configuring the webhook, trigger a workflow run in your repository. Check the collector logs for incoming webhook events. If you have the `debug` exporter enabled, you should see trace spans appearing for the workflow run, jobs, and steps.

You can also check the **Recent deliveries** section in your webhook or GitHub App settings to confirm events are being sent successfully.

## Deterministic IDs

The collector generates deterministic IDs for traces and spans, allowing you to link your own OpenTelemetry spans to the CI trace. This is useful for correlating application telemetry (e.g. deployment verification, integration tests) with the CI workflow that triggered it.

### How IDs are generated

- **Trace ID** — SHA-256 hash of `{run_id}{run_attempt}t`, truncated to 32 hex characters
- **Job Span ID** — SHA-256 hash of `{run_id}{run_attempt}{job_name}`, truncated to 16 hex characters
- **Step Span ID** — SHA-256 hash of `{run_id}{run_attempt}{job_name}{step_name}`, truncated to 16 hex characters

### Generating IDs in bash

Use these functions in your GitHub Actions workflow to compute the deterministic IDs:

```bash
generate_trace_id() {
  local run_id=$1
  local run_attempt=$2
  echo -n "${run_id}${run_attempt}t" | openssl dgst -sha256 | sed 's/^.* //' | cut -c-32
}

generate_job_span_id() {
  local run_id=$1
  local run_attempt=$2
  local job_name=$3
  echo -n "${run_id}${run_attempt}${job_name}" | openssl dgst -sha256 | sed 's/^.* //' | cut -c-16
}

generate_step_span_id() {
  local run_id=$1
  local run_attempt=$2
  local job_name=$3
  local step_name=$4
  echo -n "${run_id}${run_attempt}${job_name}${step_name}" | openssl dgst -sha256 | sed 's/^.* //' | cut -c-16
}

# Use GitHub Actions environment variables
trace_id=$(generate_trace_id ${GITHUB_RUN_ID} ${GITHUB_RUN_ATTEMPT})
job_span_id=$(generate_job_span_id ${GITHUB_RUN_ID} ${GITHUB_RUN_ATTEMPT} ${GITHUB_JOB})
step_span_id=$(generate_step_span_id ${GITHUB_RUN_ID} ${GITHUB_RUN_ATTEMPT} ${GITHUB_JOB} "your-step-name")
```

You can adapt this logic to any language that supports SHA-256 hashing. The key is matching the input string format used by the receiver.
