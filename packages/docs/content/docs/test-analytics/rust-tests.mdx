---
title: Rust Tests
description: Set up test analytics for Rust projects that use cargo test.
---

Everr parses standard `cargo test` output to track individual Rust test results, detect flaky tests, and analyze failures.

## Setup

Run your Rust tests with the default human-readable output:

```bash
cargo test
```

### GitHub Actions workflow example

```yaml
- name: Run tests
  run: cargo test
```

Everr automatically detects and parses standard `cargo test` output from workflow logs.

## What gets parsed

The parser recognizes the standard Rust libtest output patterns:

| Pattern | Meaning |
|---------|---------|
| `Running unittests src/lib.rs (target/debug/deps/my_crate-...)` | Starts a new Rust test target |
| `test my_module::tests::works ... ok` | Test passed |
| `test my_module::tests::fails ... FAILED` | Test failed |
| `test my_module::tests::ignored_case ... ignored` | Test skipped/ignored |
| `Doc-tests my_crate` | Starts doctest output for a crate |

## Module hierarchy

Rust test names use the `::` separator to represent module nesting. For example:

```text
test parser::tests::parses_config ... ok
```

This is parsed as:
- **Package**: `my_crate` or the current cargo test target
- **Module path**: `parser::tests`
- **Test name**: `parses_config`

Everr synthesizes module nodes from the `::` hierarchy, so modules can be browsed like suites. Module nodes get `everr.test.is_suite=true`, nested entries get `everr.test.is_subtest=true`, and `everr.test.parent_test` points to the parent module path.

## Generated span attributes

| Attribute | Value |
|-----------|-------|
| `everr.test.name` | Full test name (e.g. `parser::tests::parses_config`) |
| `everr.test.result` | `pass`, `fail`, or `skip` |
| `everr.test.duration_seconds` | `0` (cargo test does not emit per-test durations) |
| `everr.test.framework` | `rust` |
| `everr.test.package` | Cargo test target or doctest crate name |
| `everr.test.is_suite` | `true` for synthesized module nodes that contain child tests |
| `everr.test.is_subtest` | `true` for nested module nodes and tests |
| `everr.test.parent_test` | Parent module path |
